目標：在回應標頭加入 Link，好讓 AI Agent 能夠自動發現資源 (RFC 8288)

狀況：雖然有 Link 標頭，但裡面找不到對 AI Agent 有用的關聯類型（Relation Types）

解決方案：在你的首頁加入 Link 回應標頭，指引 AI Agent 找到有用的資源。舉例來說：用 `Link: </.well-known/api-catalog>; rel="api-catalog"` 來宣傳你的 API 目錄，或用 `Link: </docs/api>; rel="service-doc"` 指向 API 文件。具體的 Link 標頭格式可以參考 RFC 8288，IANA Link Relations 則列出了所有已註冊的關聯類型。

技能：https://isitagentready.com/.well-known/agent-skills/link-headers/SKILL.md

文件：https://www.rfc-editor.org/rfc/rfc8288, https://www.rfc-editor.org/rfc/rfc9727#section-3

目標：當 AI Agent 要求時，將 HTML 回應轉為 Markdown 格式回傳

狀況：網站目前不支援給 AI Agent 使用的 Markdown 格式

解決方案：開啟「給 AI Agent 的 Markdown」功能。這樣一來，當請求標頭帶有 `Accept: text/markdown` 時，系統會回傳 Markdown 版本的內容，而一般瀏覽器預設還是會看到 HTML。請確認回應內容的 `Content-Type` 是 `text/markdown`（如果有 `x-markdown-tokens` 的話也請一併確認）。

技能：https://isitagentready.com/.well-known/agent-skills/markdown-negotiation/SKILL.md

文件：https://developers.cloudflare.com/fundamentals/reference/markdown-for-agents/

目標：在 robots.txt 中使用 Content Signals 宣告 AI 內容的使用偏好

狀況：在 robots.txt 裡找不到 Content Signals

解決方案：在你的 robots.txt 加入 `Content-Signal` 指令，明確表達你對於 AI 訓練（ai-train）、搜尋（search）和 AI 輸入（ai-input）的偏好。例如：
`Content-Signal: ai-train=no, search=yes, ai-input=no`

文件：https://contentsignals.org/, https://datatracker.ietf.org/doc/draft-romm-aipref-contentsignals/

目標：發佈 API 目錄，實現 API 自動發現功能 (RFC 9727)

狀況：找不到 API 目錄

解決方案：建立 `/.well-known/api-catalog` 並回傳 `application/linkset+json` 格式，裡面要包含一個 `linkset` 陣列。每個項目都要有 API 的「錨點（anchor）」網址，以及相關的連結關聯，例如 `service-desc`（OpenAPI 規格）、`service-doc`（文件）和 `status`（健康檢查端點）。範例可以參考 RFC 9727 的附錄 A。

技能：https://isitagentready.com/.well-known/agent-skills/api-catalog/SKILL.md

文件：https://www.rfc-editor.org/rfc/rfc9727, https://www.rfc-editor.org/rfc/rfc9264

目標：發佈 OAuth/OIDC 發現後設資料，讓 AI Agent 知道如何串接你的 API 驗證

狀況：找不到 OAuth/OIDC 的發現後設資料

解決方案：如果你的網站有受保護的 API，請發佈 `/.well-known/openid-configuration`（給 OpenID Connect）或 `/.well-known/oauth-authorization-server`（給純 OAuth 2.0）。內容需包含發行者（issuer）、授權端點（authorization_endpoint）、憑證端點（token_endpoint）、JWKS 網址（jwks_uri）以及支援的授權類型（grant_types_supported）。這能讓 AI Agent 透過程式自動發現該如何進行身份驗證。

技能：https://isitagentready.com/.well-known/agent-skills/oauth-discovery/SKILL.md

文件：http://openid.net/specs/openid-connect-discovery-1_0.html, https://www.rfc-editor.org/rfc/rfc8414

目標：發佈 OAuth 受保護資源的後設資料，方便 AI Agent 了解驗證方式

狀況：找不到 OAuth 受保護資源的後設資料

解決方案：發佈 `/.well-known/oauth-protected-resource`，並列出你的資源識別碼、授權伺服器（可以為此資源發放憑證的 OAuth/OIDC 發行者網址清單）以及支援的範圍（scopes_supported）。這會告訴 AI Agent 該去哪裡取得存取受保護 API 的 Token。

技能：https://isitagentready.com/.well-known/agent-skills/oauth-discovery/SKILL.md

文件：https://www.rfc-editor.org/rfc/rfc9728

目標：發佈 AI Agent 技能發現索引

狀況：找不到技能發現索引

解決方案：按照 Agent Skills Discovery RFC v0.2.0 的規範，在 `/.well-known/agent-skills/index.json` 發佈索引。內容需包含 `$schema` 欄位，以及一個 `skills` 陣列，每個項目要有名稱、類型、描述、網址和 sha256 雜湊值。

目標：支援 WebMCP，透過瀏覽器將網站工具開放給 AI Agent 使用

狀況：頁面載入時沒有偵測到 WebMCP 工具

解決方案：實作 WebMCP API。呼叫 `navigator.modelContext.provideContext()` 並傳入工具定義，將網站的核心功能開放給 AI Agent 使用。每個工具都要有名稱、描述、輸入架構（JSON Schema）以及一個執行回呼函數（execute callback function）。

技能：https://isitagentready.com/.well-known/agent-skills/webmcp/SKILL.md

文件：https://webmachinelearning.github.io/webmcp/, https://developer.chrome.com/blog/webmcp-epp
