## CTO 3.0 如何採用微前端框架

雖然 CTO 3.0 本身為單一系統，但考慮到未來如果想擴充其他功能系統，建議可以使用微前端架構設計系統
即使目前僅有一個系統模組，也可以先建立「可擴充的主系統框架」，讓 CTO 3.0 本身成為第一個子系統來做使用，未來如果還有其他系統就可以整併進這套微前端框架，優點是各個系統都可以獨立維護、測試、部署，並與基底框架 Shell 溝通，避免在單一專案的架構下，遇到功能模組越來越多，導致測試、上線會彼此牽制的情境，預想架構如下：

主系統 SPA（Shell 層 / 集團入口）
├── 身份驗證（Entra ID / SSO）
│   ├── 登入 / 續期（更新 token）
│   └── 登出（清除身份狀態）
├── 身份與權限（Session：token / user / permissions）
├── UI 框架（Topbar / Sidebar / Layout）
├── 路由與導頁（統一跨系統導頁）
├── 微前端容器（qiankun：載入 / 卸載 / 切換子系統）
│   ├── 整頁型子系統（各部門子 SPA）
│   │   ├── A 部門子系統 SPA
│   │   │   └── 使用 Shell 傳遞的身份
│   │   ├── B 部門子系統 SPA
│   │   │   └── 使用 Shell 傳遞的身份
│   │   └── C 部門子系統 SPA
│   │       └── 使用 Shell 傳遞的身份
│   └── 嵌入型區塊（同頁整合跨系統功能）
│       ├── 跨系統功能共用容器
│       └── 其他可嵌入業務區塊
└── （其餘 Shell 層能力）

也就是說主系統本身不承載業務邏輯，只處理系統入口、視覺一致性、權限管理與子系統渲染判斷相關等等偏向全域設定的邏輯。

---

子系統 SPA（部門系統）
├── 業務功能與流程（該部門負責的業務）
│   ├── 業務畫面與頁面
│   ├── 業務邏輯與流程
│   └── 與後端 API 對接
├── 接收 Shell 身份（由主系統傳入，不二次登入）
│   ├── 接收 token / user
│   └── 權限判斷（依 Shell 傳入的 permissions）
├── 對接 Portal 能力（需調用他系統時，經由主系統介面）
│   ├── 呼叫主系統介面（如：打開他系統簽核、選擇組織等）
│   └── 接收主系統回傳結果
├── 獨立部署與技術選型
│   ├── 建置與部署（子系統獨立產出、獨立上線）
│   └── 技術棧選型（可依部門選用 Vue / React 等，由主系統載入）
└── 實作結構（對應程式目錄，供開發對照）
    ├── 畫面與頁面（對應頁面/畫面元件）
    ├── 共用元件與資源（對應共用元件、靜態資源）
    ├── 狀態與路由（對應狀態管理、路由設定）
    ├── 對外介面（生命週期導出、Portal API 呼叫處）
    └── 與後端對接（對應 API 呼叫、後端串接）

只需要與主系統達成溝通協議以及被定義生命週期，其他所有業務邏輯、技術選用、版本控管等等都是在該專案下完成的，可以理解成把 CTO 3.0 這個完整的系統嵌入進主系統框架。

## 未來如何與 MiTAC 其他系統整合微前端框架

如果未來集團想要把各個部門的系統整併在一起，做成統一入口或儀表板，可以在現有主系統的基礎上，進一步擴充 Shell 層的職責：
不僅提供 sidebar / topbar 加上一個主內容區，而是為每一個「集團級路由」定義要渲染的子系統，並在 Shell 層集中處理身份驗證、權限與跨系統導頁。原本已經存在、獨立開發的系統，只需要調整與主系統溝通的參數與生命週期，即可被掛載進微前端框架中，相比單一大專案，更具彈性且更好維護。

並且在這樣的整合架構下，也能大幅減少「在 A 系統中想要使用 B 系統功能畫面」的痛點。原先以 iframe 嵌入其他系統，除了要自行處理跨域下的身份驗證傳遞、cookie 與 token 同步等問題，也受限於 iframe 的隔離特性，無法與主頁面共用路由、狀態或元件，難以發揮現代前端框架的整合優勢；改為微前端架構後，由主系統統一處理登入與權限，子系統在同一頁面上下文中載入。跨子系統溝通的大致流程如下：

1. **子系統 A** 透過主系統提供的 Portal API，發送「要調用子系統 B 的某個功能」的請求（附帶必要參數）。
2. **主系統** 收到後，在「跨系統功能共用容器」（如 Modal / Drawer）中載入並渲染子系統 B 的對應畫面，並傳入身份與參數。
3. **子系統 B** 在嵌入模式下完成操作後，將結果回傳給主系統。
4. **主系統** 關閉共用容器，並將結果轉交給子系統 A，由 A 依結果更新畫面。

整體由主系統擔任中介，子系統之間不直接通訊，權限與流程皆在 Shell 層統一控管。
