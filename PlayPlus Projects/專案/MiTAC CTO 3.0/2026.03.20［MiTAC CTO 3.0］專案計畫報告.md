## **PlayPlus 普魯士國際** {#playplus-普魯士國際}

普魯士國際成立於 2013 年，以 Technology 與 Digital 為核心的設計與系統開發團隊。

我們的定位並非單純執行任務或寫程式的代工，而是**從實務與商業角度出發、協助客戶規劃真正適合的數位解決方案**的顧問。

### **幫客戶多想一步**

* 不只完成需求，而是理解需求背後的商業目標。  
* 不只交付畫面，而是確保系統能長期使用與擴充。  
* 不只著眼當下版本，而是思考未來演進的可能性。

### **深耕系統領域**

近年來，我們從品牌官網與電商等形象類型專案，轉向深耕**企業內部系統**與**複雜流程型的平台**，因為在這種比較複雜的操作情境，更能夠發揮我們的價值，為受眾設計更好的使用者體驗。

通常**企業內部系統**包含多元角色權限（例如業務、技術、客戶、管理者）、跨系統整合、流程橫跨多部門，並且需要長期維運及優化，這類型專案的思維與 CTO 高度相似。

## **我們對專案的理解　　　** {#我們對專案的理解　　　}

我們將 CTO 3.0 視為一個「正在轉型中的銷售支援平台」，不只是單純的配置產品規格及組成報價單，應該是多位角色彼此配合的情境，達成「支援銷售」的目標。

| 命名的一致性 | 目前看到的產品命名不一致，增加溝通成本，我們會讓系統改為抓取標準資料來源，再整理成適合使用者閱讀的格式。 |
| :---- | :---- |
| **需要降低挫折感** | 當規格全部選完後才提出錯誤，會讓使用者感到挫折與疲倦。我們可以即時檢核、即時提示，在當下就讓使用者知道是否符合規則。提示訊息不只告知錯誤，還要指引下一步，所以步驟腳本設計很重要，做得好，能夠大幅降低挫折感。 |
| **更容易閱讀的關鍵規格** | 現在需逐一點進產品內頁才能看得到規格，依靠記憶力進行比對。我們可以在清單頁面條列式顯示關鍵規格，幫助使用者快速理解差異。 |
| **規格比對** | 再新增一個頁面專門支援產品比對，讓 CTO 從單純填寫資料的工具再更往輔助銷售靠近。 |

| **後端獨立部署** | 將後端邏輯抽離於原本的資料庫，讓邏輯判斷能夠依照自己的想法調整。 |
| **業務人員的報價管理** | 新構想，可以為多位客戶、多個專案儲存報價方案，進行情境式的銷售提案。 |

## **設計　　　** {#設計　　　}

設計是本專案的啟動引擎，我們不單是美化介面，而是透過設計解決現有系統在溝通、操作與決策上的斷點。

### **語義與資訊的重新對口**

* **消除溝通鴻溝**： 將技術導向的 P-code 與料號，在 UI 層級轉譯為業務與客戶聽得懂的商品語言。透過設計「標籤系統」與「動態提示」，確保 CTO 平台的資料與 Zoho CRM、Sales Kit 保持一致，減少對照時間。  
* **資訊優先級定義**： 重新梳理資訊架構（Information Architecture），確保使用者在配置過程中，目光第一時間能停留在「影響決策」的關鍵參數上。

### **回饋機制的即時化與正向化**

* **防呆即是引導**： 捨棄「事後檢查」的傳統模式，設計「即時驗證規則」。當使用者選擇了一個不相容的組件，UI 會立即給出溫馨提示，並「主動建議」可行的替代方案，將挫折感轉化為引導感。  
* **步驟式導航設計**： 針對複雜的配置流程，設計清晰的進度指標與腳本式引導，讓使用者清楚知道「我現在在哪」以及「下一步該做什麼」。

### **從「資料填寫」轉向「輔助銷售」**

* **視覺化規格比較**： 開發專屬的「報價對比」介面，讓業務能直接呈現不同配置方案的優劣與價格差異，提升說服力，而不僅僅是產出 PDF。  
* **動態價格感知**： 設計透明的價格異動顯示機制，讓使用者在調整參數時，能感知到價值與成本的關聯，優化議價過程。

### **智慧化與互動性的深度融合**

* **無縫 AI 嵌入**： 將集團 AI 工具（如 ChatBot）自然融合於操作動線中，不只是側邊欄位，而是能根據當前配置狀態提供「智慧建議」與「一鍵查詢」。  
* **狀態感知介面**： 針對與 Zoho CRM 和 WOM 系統的串接，設計更直覺的「報價生命週期」視覺化，讓業務一眼就能看出哪些報價正處於「等待回覆」、「已核准」或「需修正」狀態。

## **前端　　　** {#前端　　　}

我們選擇最適合 MiTAC 的前端開發方式，考慮開發效率、可維護性及可擴充性。

### **CTO 前台頁面**

我們對 CTO 前台頁面的前端規劃不僅止於資料搜集，更強調「銷售輔助」的任務定位。

* **語義化轉譯與資訊對齊**： 前端將艱澀的 P-code 與技術料號轉譯為業務與客戶看得懂的商品語言，並確保資料標籤與 Zoho CRM 及 Sales Kit 保持一致。  
* **防呆即時回饋**： 規則即時校驗機制。當使用者選擇不相容組件時，系統不只是顯示錯誤，而是主動建議可行的替代方案，將操作過程中的挫折感轉化為流暢的引導。  
* **視覺化規格比較**： 開發專屬的比對頁面，讓業務能直接呈現不同配置方案的差異。  
* **智慧化 AI 深度嵌入**： 將集團 AI 工具自然融合於操作動線中，提供「智慧建議」與「一鍵查詢」等功能，提升自助配置的成功率。

### **管理者後台**

為了降低系統長期維護成本，以及增加自主控制能力，我們將量身打造管理控制台。

* **無代碼規則引擎管理**： 從管理介面定義產品組合規則與相容性限制。  
* **報價單管控**： 提供戰情室視角，即時掌控報價單的狀態（如已提交或議價中），協助主管精準掌握業務動態。  
* **權限管理**： 建立角色權限矩陣，區分管理者、主管、業務與外部客戶，確保資料存取的安全性。  
* **系統稽核與追蹤**： 詳細記錄所有規則異動與報價單修改軌跡（誰在何時改了什麼），確保異動皆可追溯。

### **Vue 3**

我們選擇 Vue 3 作為前端主開發框架，可以滿足 CTO 3.0 對於複雜邏輯與長期維護的高標準需求。

* **Composition API**： CTO 系統涉及大量的產品配置規則與互斥邏輯。透過 Vue 3 的 Composition API 可以將複雜的業務邏輯抽離成獨立且可測試的模組，避免程式碼過度耦合，大幅提升開發效率與代碼品質。  
* **流暢體驗**： Vue 3 採用底層優化的 Proxy-based 響應式系統與更細粒度的更新機制，能確保在處理包含數百個參數、即時變動的產品配置頁面時，依然保持毫秒級的回饋速度，消除使用者操作時的卡頓感。  
* **TypeScript**： 企業級系統更需要高度的穩定性。透過 Vue 3 對 TypeScript 的原生支援，讓我們在開發階段就能透過型別檢查揪出潛在錯誤，確保與後端 API 對接時的資料一致性。  
* **長期維護性**： Vue 3 能確保 MiTAC 的系統在未來 5-10 年內依然具備良好的擴充空間，且容易招募到具備相應技術背景的維運人才。

## **微前端架構　　　** {#微前端架構　　　}

此段落用於補充說明前端。考量到 MiTAC 集團未來的系統生態整合，我們將採用 qiankun 微前端框架。

雖然目前 CTO 3.0 僅是單一系統，但考量到未來 MiTAC 集團大量系統的長期規劃，現在就是導入「微前端框架」的適合時機。微前端框架內的各系統能獨立開發、測試與部署，避開傳統單體架構的互相牽制問題，未來若有新系統加入，可以無痛併入。

我們會建立一個「可擴充的主系統框架（Shell）」，並以 qiankun 作為主框架，再將 CTO 3.0 納入為第一個子系統。這樣的規劃有以下兩個核心價值。

* **支援集團未來擴充**： 日後若其他部門系統要納入集團入口，可依照統一規範接入，不需要重做整體平台。  
* **兼顧現階段專案推進效率**： CTO 3.0 仍可依自身需求獨立開發與上線，同時提前建立集團級架構基礎，讓這個專案具備延伸性。

### **![][image3]**

### **1\. 主系統 SPA（集團入口 / 系統框架層）**

這層等於是整個大系統的「基底」與「統籌中心」。它本身不負責處理特定的業務邏輯，而是專注於全域的視覺統整與流程管理。

* **統一的入口與視覺框架**： 負責全站共用的頂部導覽列（Topbar）、側邊選單（Sidebar）與整體排版（Layout），維持跨系統的一致體驗。  
* **統一的身分驗證與權限派發**： 整合登入系統（如 Entra ID / SSO），處理 Session 與 Token 的狀態。由主系統統一管理「誰登入」、「有哪些權限」。  
* **路由與跨系統導航**： 負責告訴使用者目前應該看到哪個系統，並管理跨系統的頁面切換。  
* **微前端容器管理**： 負責決定什麼時候載入子系統、何時將子系統卸載或切換。

### **2\. 子系統 SPA（部門業務系統）**

CTO 3.0 將作為系統內的第一個獨立子系統，只需專注在自己的業務流程上，不需要煩惱登入或跨系統溝通的底層架構。

* **專注核心業務**： 處理報價、產品配置等專屬該系統的所有畫面與業務邏輯。  
* **繼承主系統身分**： 由主系統傳入登入狀態與權限指標，使用者在不同子系統間不需重複登入。  
* **完全獨立的開發與上線步調**： 各部門系統可以依照自己的時程排程獨立產出、上線。並且在技術選型上高度自由，不管未來新系統使用 Vue 或 React，只要符合主系統接入規範，皆可無縫掛載。

### **為何選擇 qiankun 而非 MicroAPP**

MicroAPP 的優點是接入輕量、初期整合快速，適合單點導入或較鬆散的系統整合情境。但如果目標是打造集團統一入口平台，則 qiankun 更適合作為主框架，原因在於 qiankun 對於 子系統生命週期管理、主子應用協作規範、路由整合、權限傳遞與統一治理 都更完整，較適合未來需要持續擴編、跨部門整合、長期維運的大型系統環境。

#### **1\. 接入模型**

MicroAPP 採用「容器嵌入式」，主應用以標籤指定子應用 URL 即可載入，子應用僅需可獨立運行並提供載入位址，無須配合主應用撰寫註冊邏輯。對既有系統、外包建置、或各部門技術棧不一致的情境，改動少、接入成本低。  
qiankun 採「主應用註冊式」，主應用須為每個子應用定義 name 、 entry 、 container 、 activeRule 、 props 並可選配 lifecycle hooks。當子系統想要納入平台，須符合此定義（調整 entry 與導出生命週期），雖然接入門檻較高，但主應用對子應用掛載時機與範圍有完整掌控。

#### **2\. 路由治理能力**

MicroAPP 著重於將異質系統嵌於同一入口，路由與子應用啟用條件多由各子應用或團隊自行約定，主應用不強制規範。  
qiankun 以 URL 規則驅動子應用啟用，而 activeRule 支援字串、函式、陣列，便於實作集團入口常見的「依路徑分域」策略，例如「 /hr/\* 」、「 /sales/\* 」、「 /erp/\* 」或「 /finance/\* 」。  
qiankun 對「集團首頁 → 各部門 → 各業務子平台」此類層級式路由可明確劃分責任邊界，利於路由治理。

#### **3\. 生命週期與接入契約**

MicroAPP 支援整合、通訊與事件，但生命週期與接入契約無強制規範，管理時點與檢查流程須由團隊自行訂定。  
qiankun 將 lifecycle 列為主應用的一級能力，提供 beforeLoad 、 beforeMount 、 afterMount 、 beforeUnmount 、 afterUnmount 等 hooks，主應用可於平台層統一注入邏輯。此設計直接支援子系統納管流程、上線前檢查、錯誤定位、載入稽核與平台維運一致性，易於訂定標準作業程序。

#### **4\. 主應用作為平台治理中心**

MicroAPP 定位為高彈性的整合殼層，主應用雖可執行管理，但平台規範、治理機制與各階段職責分工，須另行建立與維護。  
qiankun 於 API 層即提供 registerMicroApps（統一註冊） 、 props（主應用主動下發資料） 、 initGlobalState（全域狀態） 、 prefetchApps（預載） 、 addGlobalUncaughtErrorHandler（全域錯誤處理） 、 runAfterFirstMounted（首個子應用掛載完成後之平台動作）。想要將主應用作為集團級平台治理中心時，可直接運用上述能力，無須自建底層機制。

#### **5\. 上下文傳遞**

MicroAPP 通訊方式彈性，惟 token 或角色權限等上下文之傳遞與維護須由團隊明確定義，否則易出現各系統實作方式不一、不利統一管理之情形。  
qiankun 由主應用透過 props 與 initGlobalState 主動發放共用上下文至子應用，傳遞路徑固定，易於形成契約並要求各部門遵循。

#### **6\. 樣式隔離與平台穩定性**

qiankun 於 start / loadMicroApp 之設定中提供 sandbox 、 strictStyleIsolation 、 experimentalStyleIsolation 並可搭配 Shadow DOM 強化樣式隔離；官方文件亦說明部分 CSS 規則（如 @keyframes 或 @font-face 或 @import 或 @page）於沙盒下之限制。於多部門系統並存之大型平台，JS 沙盒、樣式隔離、多系統共存時之污染控制，以及子系統卸載與切換之穩定性，qiankun 皆提供可配置選項。

### **未來如何與 MiTAC 其他系統整合微前端框架**

未來若集團期望將各部門系統整合「統一入口」或「概況儀表板」，這套微前端架構將能派上大用場。相較於將所有功能塞在一起的巨型單一專案，微前端具備更好的可擴充性。

我們不需打掉重練，只需擴充主系統（Shell）的職責範圍，根據不同的群組與路由直接掛載對應的子系統。原本各自為政的獨立專案，只要稍微調整與主系統的溝通接口（參數與生命週期）就能整合。

再者，微前端架構能有效解決傳統「iframe 嵌入」的痛點。過去要在 A 系統中嵌入 B 系統的畫面，常會遇到身分驗證傳遞困難（Cookie 與 Token 不同步），加上 iframe 先天的封閉隔離特性，難以共用路由狀態或元件。改為微前端架構後，由主系統中介處理登入與權限，所有子系統都在同一個頁面環境中載入與運作，跨系統的溝通流程會變得非常流暢。

1. **提出需求**： 透過主系統提供的 Portal API，子系統 A 發送「調用子系統 B 的某個功能」的請求（附帶必要參數）。  
2. **居中調度**： 主系統收到請求後，在共用容器（如彈出式 Modal 或側拉 Drawer 介面）中載入並渲染子系統 B 的對應畫面，並附上必要的權限與參數。  
3. **完成回傳**： 嵌入模式內的子系統 B 在區塊內讓使用者完成操作後，將結果回傳給主系統。  
4. **結果交接**： 主系統關閉共用容器，子系統 A 從主系統收到完整結果，並更新自己的畫面。

總結來說，整個整合體驗是由主系統擔任堅實的「指揮中介」，子系統彼此不需複雜地相互綁定，權限與流程皆在最上層統一且安全地控管。

## **後端　　　** {#後端　　　}

除了介面設計與前端開發，我們還能夠開發穩定、安全且具備高度業務邏輯正確性的核心引擎。依據 MiTAC 的需求將採用 C\# 作為主要開發語言，並針對前端需求規畫開發重點。

### **核心運算**

後端將承載系統最關鍵的產品配置邏輯。當使用者在前端選擇料號或 P-code 時，後端 API 會即時接收數據並執行即時運算。規則引擎會根據預設的相容性與互斥邏輯，檢驗當前配置是否合法，並動態決定流程是否允許繼續進行，從底層確保每一筆報價單的精準度。

### **高效能 API 接口與前端功能支援**

配合前端規劃的直覺化體驗，後端將開發一系列高性能的 API 接口。

* **配置引導**： 提供智慧建議所需的資料庫檢索邏輯。  
* **規格對比**： 支援前端多產品規格橫向比對的資料查詢。  
* **即時狀態**： 即時更新與追蹤報價單的生命週期狀態，確前後端資料同步。  
* **權限判斷**： 依據登入角色（業務、主管、客戶）決定對應的資料存取範圍與操作權限。

### **企業級資安與 Azure SSO 整合**

* **Azure SSO**： 採用 MiTAC 現有的 Microsoft Entra ID 進行單一登入（SSO）。  
* **Token 安全驗證**： 所有 API 請求皆需通過 Token 驗證，並結合 RBAC （角色基礎存取控制）矩陣，確保僅具備權限的人員能存取敏感的報價與客戶資料。

### **第三方系統深度整合**

* **Zoho CRM 雙向同步**：串接 Zoho CRM 撈取員工及客戶重要資料，同時處理「議價單」的雙向狀態同步，讓業務在系統內即可掌握最新議價進度。  
* **WOM 訂單自動拋轉**：當報價單確認成交後，後端將自動將資料封裝並穩定地拋轉至 WOM 系統進行下單操作，達成從報價到下單的自動化閉環。

## **伺服器部署與維運策略　　　** {#伺服器部署與維運策略　　　}

專業的部署流程能確保系統健康運作與零停機更新。

* **雲端基礎建設**： 規劃建置分流環境，包含開發環境（Dev）、測試環境（UAT）及正式上線環境（Production）。  
* **CI/CD 自動化部署**： 導入如 GitHub Actions 或 Azure DevOps 等工具，所有程式碼提交需經過自動化單元測試與原始碼掃描，驗證無誤後自動封裝並部署至伺服器。  
* **資安掃描**： 採用 MiTAC 集團慣用的 SonarQube 或其他掃描工具，原始碼每次上架前需進行完整掃描，確認沒有問題才可以上架。  
* **即時追蹤狀況**： 架設 APM 工具或錯誤追蹤系統，當伺服器發生異常或前端頁面崩潰，開發與維運團隊將於第一時間收到警報，主動排除問題。  
* **安全防護與災難復原**： 設定資料庫定時備份，確保業務資料萬無一失。

C\# 後端系統開發，打造高穩定性、高安全性的運算層，支援前端複雜的業務邏輯並負責多系統間的深度整合。這個階段能夠和前端工程的時程有部分重疊，確認邏輯後即可開始同步進行。
