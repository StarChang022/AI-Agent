## CTO 3.0 如何採用微前端框架

雖然目前 CTO 3.0 僅是單一系統，但考量到未來 MiTAC 集團大量系統的長期規劃，現在就是導入「微前端框架」的適合時機。微前端框架內的各系統能獨立開發、測試與部署，避開傳統單體架構的互相牽制問題，未來若有新系統加入，可以無痛併入。

我們會建立一個「可擴充的主系統框架（Shell）」，並以 qiankun 作為主框架，再將 CTO 3.0 納入為第一個子系統。這樣的規劃有以下兩個核心價值。

1. **支援集團未來擴充**： 日後若其他部門系統要納入集團入口，可依照統一規範接入，不需要重做整體平台。
2. **兼顧現階段專案推進效率**： CTO 3.0 仍可依自身需求獨立開發與上線，同時提前建立集團級架構基礎，讓這個專案具備延伸性。

### 1. 主系統 SPA（集團入口 / 系統框架層）

這層等於是整個大系統的「基底」與「統籌中心」。它本身不負責處理特定的業務邏輯，而是專注於全域的視覺統整與流程管理。

- **統一的入口與視覺框架**： 負責全站共用的頂部導覽列（Topbar）、側邊選單（Sidebar）與整體排版（Layout），維持跨系統的一致體驗。
- **統一的身分驗證與權限派發**： 整合登入系統（如 Entra ID / SSO），處理 Session 與 Token 的狀態。由主系統統一管理「誰登入」、「有哪些權限」。
- **路由與跨系統導航**： 負責告訴使用者目前應該看到哪個系統，並管理跨系統的頁面切換。
- **微前端容器管理**： 負責決定什麼時候載入子系統、何時將子系統卸載或切換。

### 2. 子系統 SPA（部門業務系統）

CTO 3.0 將作為系統內的第一個獨立子系統，只需專注在自己的業務流程上，不需要煩惱登入或跨系統溝通的底層架構。

- **專注核心業務**： 處理報價、產品配置等專屬該系統的所有畫面與業務邏輯。
- **繼承主系統身分**： 由主系統傳入登入狀態與權限指標，使用者在不同子系統間不需重複登入。
- **完全獨立的開發與上線步調**： 各部門系統可以依照自己的時程排程獨立產出、上線。並且在技術選型上高度自由，不管未來新系統使用 Vue 或 React，只要符合主系統接入規範，皆可無縫掛載。

## 為何選擇 qiankun 而非 MicroAPP

MicroAPP 的優點是接入輕量、初期整合快速，適合單點導入或較鬆散的系統整合情境。但如果目標是打造集團統一入口平台，則 qiankun 更適合作為主框架，原因在於 qiankun 對於 子系統生命週期管理、主子應用協作規範、路由整合、權限傳遞與統一治理 都更完整，較適合未來需要持續擴編、跨部門整合、長期維運的大型系統環境。

## 簡介 qiankun

### 1. 接入模型

MicroAPP 採用「容器嵌入式」，主應用以標籤指定子應用 URL 即可載入，子應用僅需可獨立運行並提供載入位址，無須配合主應用撰寫註冊邏輯。對既有系統、外包建置、或各部門技術棧不一致的情境，改動少、接入成本低。

qiankun 採「主應用註冊式」，主應用須為每個子應用定義 name 、 entry 、 container 、 activeRule 、 props 並可選配 lifecycle hooks。當子系統想要納入平台，須符合此定義（調整 entry 與導出生命週期），雖然接入門檻較高，但主應用對子應用掛載時機與範圍有完整掌控。

### 2. 路由治理能力

MicroAPP 著重於將異質系統嵌於同一入口，路由與子應用啟用條件多由各子應用或團隊自行約定，主應用不強制規範。

qiankun 以 URL 規則驅動子應用啟用，而 activeRule 支援字串、函式、陣列，便於實作集團入口常見的「依路徑分域」策略，例如「 /hr/* 」、「 /sales/* 」、「 /erp/* 」或「 /finance/* 」。

qiankun 對「集團首頁 → 各部門 → 各業務子平台」此類層級式路由可明確劃分責任邊界，利於路由治理。

### 3. 生命週期與接入契約

MicroAPP 支援整合、通訊與事件，但生命週期與接入契約無強制規範，管理時點與檢查流程須由團隊自行訂定。

qiankun 將 lifecycle 列為主應用的一級能力，提供 beforeLoad 、 beforeMount 、 afterMount 、 beforeUnmount 、 afterUnmount 等 hooks，主應用可於平台層統一注入邏輯。此設計直接支援子系統納管流程、上線前檢查、錯誤定位、載入稽核與平台維運一致性，易於訂定標準作業程序。

### 4. 主應用作為平台治理中心

MicroAPP 定位為高彈性的整合殼層，主應用雖可執行管理，但平台規範、治理機制與各階段職責分工，須另行建立與維護。

qiankun 於 API 層即提供 registerMicroApps（統一註冊） 、 props（主應用主動下發資料） 、 initGlobalState（全域狀態） 、 prefetchApps（預載） 、 addGlobalUncaughtErrorHandler（全域錯誤處理） 、 runAfterFirstMounted（首個子應用掛載完成後之平台動作）。想要將主應用作為集團級平台治理中心時，可直接運用上述能力，無須自建底層機制。

### 5. 上下文傳遞

MicroAPP 通訊方式彈性，惟 token 或角色權限等上下文之傳遞與維護須由團隊明確定義，否則易出現各系統實作方式不一、不利統一管理之情形。

qiankun 由主應用透過 props 與 initGlobalState 主動發放共用上下文至子應用，傳遞路徑固定，易於形成契約並要求各部門遵循。

### 6. 樣式隔離與平台穩定性

qiankun 於 start / loadMicroApp 之設定中提供 sandbox 、 strictStyleIsolation 、 experimentalStyleIsolation 並可搭配 Shadow DOM 強化樣式隔離；官方文件亦說明部分 CSS 規則（如 @keyframes 或 @font-face 或 @import 或 @page）於沙盒下之限制。於多部門系統並存之大型平台，JS 沙盒、樣式隔離、多系統共存時之污染控制，以及子系統卸載與切換之穩定性，qiankun 皆提供可配置選項。

## 未來如何與 MiTAC 其他系統整合微前端框架

未來若集團期望將各部門系統整合「統一入口」或「概況儀表板」，這套微前端架構將能派上大用場。相較於將所有功能塞在一起的巨型單一專案，微前端具備更好的可擴充性。

我們不需打掉重練，只需擴充主系統（Shell）的職責範圍，根據不同的群組與路由直接掛載對應的子系統。原本各自為政的獨立專案，只要稍微調整與主系統的溝通接口（參數與生命週期）就能整合。

再者，微前端架構能有效解決傳統「iframe 嵌入」的痛點。過去要在 A 系統中嵌入 B 系統的畫面，常會遇到身分驗證傳遞困難（Cookie 與 Token 不同步），加上 iframe 先天的封閉隔離特性，難以共用路由狀態或元件。改為微前端架構後，由主系統中介處理登入與權限，所有子系統都在同一個頁面環境中載入與運作，跨系統的溝通流程會變得非常流暢。

1. **提出需求**： 透過主系統提供的 Portal API，子系統 A 發送「調用子系統 B 的某個功能」的請求（附帶必要參數）。
2. **居中調度**： 主系統收到請求後，在共用容器（如彈出式 Modal 或側拉 Drawer 介面）中載入並渲染子系統 B 的對應畫面，並附上必要的權限與參數。
3. **完成回傳**： 嵌入模式內的子系統 B 在區塊內讓使用者完成操作後，將結果回傳給主系統。
4. **結果交接**： 主系統關閉共用容器，子系統 A 從主系統收到完整結果，並更新自己的畫面。

總結來說，整個整合體驗是由主系統擔任堅實的「指揮中介」，子系統彼此不需複雜地相互綁定，權限與流程皆在最上層統一且安全地控管。
