## PlayPlus 普魯士國際

普魯士國際成立於 2013 年，以 **Technology** 與 **Digital** 為核心的設計與系統開發團隊。

我們的定位並非單純執行任務或寫程式的代工，而是**從實務與商業角度出發、協助客戶規劃真正適合的數位解決方案**的顧問。

### 幫客戶多想一步

團隊核心理念是「**幫客戶多想一步**」。

- 不只完成需求，而是理解需求背後的商業目標。
- 不只交付畫面，而是確保系統能長期使用與擴充。
- 不只著眼當下版本，而是思考未來演進的可能性。

### 深耕系統領域

近年來，我們從品牌官網與電商等形象類型專案，轉向深耕**企業內部系統**與**複雜流程型的平台**，因為在這種比較複雜的操作情境，更能夠發揮我們的價值，為受眾設計更好的使用者體驗。

通常**企業內部系統**包含多元角色權限（例如業務、技術、客戶、管理者）、跨系統整合、流程橫跨多部門，並且需要長期維運及優化，這類型專案的思維與 CTO 高度相似。

## 我們對專案的理解

我們將 CTO 3.0 視為一個「**正在轉型中的銷售支援平台**」，不只是單純的配置產品規格及組成報價單，應該是多位角色彼此配合的情境，達成「支援銷售」的目標。

以下是現況遭遇的挑戰，及我們打算如何改變。

1. 命名不一致，增加溝通成本
    
    
    | 現況 | 現在的 CTO 以 P-code、料號為主，與官網、Sales Kit、業務慣用語言不一致導致業務或客戶需要額外對照，降低銷售效率。 |
    | --- | --- |
    | 目標 | 讓系統改為抓取標準資料來源，例如 Zoho CRM 的資料如果是公司內部的標準，就統一從這邊抓取資料，再後續整理成適合使用者閱讀的格式。 |
2. 配置錯誤的回饋過晚，容易造成挫折感
    
    
    | 現況 | 規格全部選完後才檢查規則，這時候提出錯誤，會讓使用者感到挫折。 |
    | --- | --- |
    | 目標 | 我們可以即時檢核、即時提示，在當下就讓使用者知道是否符合規則。提示訊息不只告知錯誤，還要指引下一步，所以步驟腳本設計很重要，做得好，能夠大幅降低挫折感。 |
3. 規格比對不直覺，難以輔助決策
    
    
    | 現況 | 現在需逐一點進產品內頁才能看得到規格，自行依靠記憶力比對規格，無法幫助業務與客戶快速理解差異。 |
    | --- | --- |
    | 目標 | 我們可以在清單頁面條列式顯示關鍵規格，**再新增一個頁面專門支援產品比對**。讓 CTO 從單純填寫資料的工具再更往輔助銷售靠近。 |
4. 善用 AI 科技
    
    
    | 現況 | CTO 尚未應用 MiTAC 的 AI 工具。 |
    | --- | --- |
    | 目標 | 將集團內部開發的聊天機器人或 AI 模型，應用至系統即時客服或資料查詢，亦可在規格配置過程中提供建議。 |
5. 加強 CTO 與其他系統的互動
    
    CTO 流程後段需要組成報價單提交至 Zoho CRM 等待報價，從 Zoho CRM 拿回報價金額後再操作決定是否接受報價，這就是一來一回的議價過程。接受報價後，再將報價單轉拋至 WOM 系統正式下訂單。
    
    目前尚未得知具體需要改善的地方，但可以想像到，如果能夠在 CTO 這邊增加狀態追蹤（例如正在等待報價或已報價）、通知提醒（例如新報價）、回覆內容（兩邊系統可以新增備註訊息來往）等措施，將大大幫助增加互動性。
    
6. 業務人員的報價管理
    
    此為新想法，可以為多位客戶、多個專案儲存報價方案，進行情境式的銷售提案。

## 時程規劃

以下以「工作天」為單位，說明每個階段的實際工作內容與目的。預計總共 140-160 個工作天。

### 階段1. 訪談

`10 個工作天`

收集細節資料，幫助我們更理解專案，以及確認雙方共識，理想狀況是收集更多使用者對於 CTO 2.0 的回饋。

- CTO 2.0 實際使用痛點。
- 哪些地方「功能合理，但實際操作很卡」。
- 了解每個流程實際應該要產出什麼結果。

### 階段2. 使用者體驗

`40 個工作天`

UX 黑白稿主要任務是梳理流程及規則，特別是 CTO 與 Zoho CRM 的確認報價互動，可能還要思考 MiTAC 的 AI 工具如何應用。

其餘流程已經在 CTO 2.0 定義的很清楚，所以會放比較多時間確認使用者在每個地方「此刻最需要知道什麼」。

### 階段3. 介面視覺

`30 個工作天`

UI 彩色稿實現具體操作介面，統一設計風格，將 UX 的草稿轉為真實應用的元件，及定義功能效果、提示、回饋等，補齊 UX 黑白稿不易表達的互動。

除此之外，其實 UI 階段還附加複測操作細節的效果，看著彩色稿會更明確思考操作行為。

### 階段4. 前端

`60 個工作天`

設計稿除了產出畫面，也會補充功能效果的說明，已經足夠讓工程團隊討論 API 規格。

所以確認定稿後（或接近定稿時），安排工程會議，同步雙方團隊的 API 想法及開發方式，前端便可開始依照設計稿進行開發。

### 測試

`40 個工作天`

實際業務操作情境，完整走過「產品配置（含錯誤情境）、報價確認、接受報價並導向 WOM」等重要流程，確認使用者是否清楚「現在在做什麼」、當下系統是否給出足夠的資訊，驗證 CTO 3.0 是否真的能支撐實際情境。

- 流程驗證
- 角色與權限測試
- 規格與錯誤提示測試
- 與既有系統的整合驗證
- 使用者測試與回饋修正
- 上線前交付準備

## 前端技術應用

我們的前端開發方式，長期專注於**系統的穩定性、可維護性與可擴展性**，像是 CTO 這類「使用年限長、角色多、需要流程」的企業級平台就是我們深耕的領域。

- **Vue 3**： 前端框架使用 Vue 3 ，他的模板語法接近 HTML，Composition API 貼近一般寫 JavaScript/TypeScript 的思維，輕量、開發快速、易於交接與維護是他的優點。
- **Vite：**開發環境下無需預先打包，以原生的 ESM 處理模組載入可以快速啟動伺服器，並且熱更新可以即時看到程式碼改變後的畫面，大幅解決程式越來越龐大導致開發速度變慢的痛點。
- **TypeScript：在**編譯階段時就會提供靜態型別檢查，減少實際執行發生的錯誤，且透過強型別語法提升程式碼的可讀性、穩定性與維護性。
- **Tailwind CSS**： 減少寫自訂CSS的時間，樣式直接寫在 template，可讀性高，可以快速建立一致且易維護的 UI 元件。
- **MicroApp：**基於 WebComponent 組件化思維的微前端框架，子應用有各自的程式庫與建置流程，可獨立開發、測試、上線，不需等主應用一起更新，且可以使用其他的技術而不影響主應用，適合未來需要擴充大型功能的系統。
