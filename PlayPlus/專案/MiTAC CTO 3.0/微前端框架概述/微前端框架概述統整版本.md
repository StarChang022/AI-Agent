## CTO 3.0 如何採用微前端框架

雖然目前 CTO 3.0 僅是單一系統，但考量到未來 MiTAC 集團大量系統的長期規劃，我們建議初期就導入「微前端框架」的系統架構設計。我們會為 CTO 3.0 建立一個「可擴充的主系統框架（Shell）」，並讓 CTO 3.0 成為架構下的「第一個子系統」。

這樣做有個最大益處，未來若有新系統加入，可以輕鬆無痛併入。微前端框架內的各系統能獨立開發、測試與部署，避開傳統單體架構的互相牽制問題。

### 1. 主系統 SPA（集團入口 / 系統框架層）

這層等於是整個大系統的「基底」與「統籌中心」。它本身**不負責處理特定的業務邏輯**，而是專注於全域的視覺統整與流程管理。

- **統一的入口與視覺框架**：負責全站共用的頂部導覽列（Topbar）、側邊選單（Sidebar）與整體排版（Layout），維持跨系統的一致體驗。
- **統一的身分驗證與權限派發**：整合登入系統（如 Entra ID / SSO），處理 Session 與 Token 的狀態。由主系統統一管理「誰登入」、「有哪些權限」。
- **路由與跨系統導航**：負責告訴使用者目前應該看到哪個系統，並管理跨系統的頁面切換。
- **微前端容器管理**：負責決定什麼時候載入子系統、何時將子系統卸載或切換。

主系統 SPA（Shell 層 / 集團入口）
├── 身份驗證（Entra ID / SSO）
│   ├── 登入 / 續期（更新 token）
│   └── 登出（清除身份狀態）
├── 身份與權限（Session：token / user / permissions）
├── UI 框架（Topbar / Sidebar / Layout）
├── 路由與導頁（統一跨系統導頁）
├── 微前端容器（qiankun：載入 / 卸載 / 切換子系統）
│   ├── 整頁型子系統（各部門子 SPA）
│   │   ├── A 部門子系統 SPA
│   │   │   └── 使用 Shell 傳遞的身份
│   │   ├── B 部門子系統 SPA
│   │   │   └── 使用 Shell 傳遞的身份
│   │   └── C 部門子系統 SPA
│   │       └── 使用 Shell 傳遞的身份
│   └── 嵌入型區塊（同頁整合跨系統功能）
│       ├── 跨系統功能共用容器
│       └── 其他可嵌入業務區塊
└── （其餘 Shell 層能力）

### 2. 子系統 SPA（部門業務系統）

CTO 3.0 將作為系統內的第一個獨立子系統，**只需專注在自己的業務流程上**，不需要煩惱登入或跨系統溝通的底層架構。

- **專注核心業務**：處理報價、產品配置等專屬該系統的所有畫面與業務邏輯。
- **繼承主系統身分**：由主系統傳入登入狀態與權限指標，使用者在不同子系統間不需重複登入。
- **完全獨立的開發與上線步調**：各部門系統可以依照自己的時程排程獨立產出、上線。並且在技術選型上高度自由，不管未來新系統使用 Vue 或 React，只要符合主系統接入規範，皆可無縫掛載。

子系統 SPA（部門系統）
├── 業務功能與流程（該部門負責的業務）
│   ├── 業務畫面與頁面
│   ├── 業務邏輯與流程
│   └── 與後端 API 對接
├── 接收 Shell 身份（由主系統傳入，不二次登入）
│   ├── 接收 token / user
│   └── 權限判斷（依 Shell 傳入的 permissions）
├── 對接 Portal 能力（需調用他系統時，經由主系統介面）
│   ├── 呼叫主系統介面（如：打開他系統簽核、選擇組織等）
│   └── 接收主系統回傳結果
├── 獨立部署與技術選型
│   ├── 建置與部署（子系統獨立產出、獨立上線）
│   └── 技術棧選型（可依部門選用 Vue / React 等，由主系統載入）
└── 實作結構（對應程式目錄，供開發對照）
    ├── 畫面與頁面（對應頁面/畫面元件）
    ├── 共用元件與資源（對應共用元件、靜態資源）
    ├── 狀態與路由（對應狀態管理、路由設定）
    ├── 對外介面（生命週期導出、Portal API 呼叫處）
    └── 與後端對接（對應 API 呼叫、後端串接）

## 未來如何與 MiTAC 其他系統整合微前端框架

未來若集團期望將各部門系統整合「統一入口」或「概況儀表板」，這套微前端架構將能派上大用場。相較於將所有功能塞在一起的巨型單一專案，微前端具備更好的可擴充性。

我們不需打掉重練，只需擴充主系統（Shell）的職責範圍，根據不同的群組與路由直接掛載對應的子系統。原本各自為政的獨立專案，只要稍微調整與主系統的溝通接口（參數與生命週期）就能整合。

再者，微前端架構能有效解決傳統「iframe 嵌入」的痛點。過去要在 A 系統中嵌入 B 系統的畫面，常會遇到身分驗證傳遞困難（Cookie 與 Token 不同步），加上 iframe 先天的封閉隔離特性，難以共用路由狀態或元件。改為微前端架構後，由主系統中介處理登入與權限，所有子系統都在同一個頁面環境中載入與運作，跨系統的溝通流程會變得非常流暢。

1. **提出需求**： 透過主系統提供的 Portal API，子系統 A 發送「調用子系統 B 的某個功能」的請求（附帶必要參數）。
2. **居中調度**： 主系統收到請求後，在共用容器（如彈出式 Modal 或側拉 Drawer 介面）中載入並渲染子系統 B 的對應畫面，並附上必要的權限與參數。
3. **完成回傳**： 嵌入模式內的子系統 B 在區塊內讓使用者完成操作後，將結果回傳給主系統。
4. **結果交接**： 主系統關閉共用容器，子系統 A 從主系統收到完整結果，並更新自己的畫面。

**總結來說，整個整合體驗是由主系統擔任堅實的「指揮中介」，子系統彼此不需複雜地相互綁定，權限與流程皆在最上層統一且安全地控管。**
