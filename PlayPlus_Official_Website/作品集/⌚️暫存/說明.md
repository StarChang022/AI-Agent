# 網站介紹

**朝朝學美** 是一個結合「營養科學」與「習慣養成」的健康管理數位平台。其核心價值在於透過日復一日的小改變，協助使用者建立正確的飲食習慣，並透過數據化的生理追蹤（體重、體脂、生理期），提供具備心理支持的減重旅程。

* **專案類型**：數位健康管理平台 / 會員制 Web App
* **目標對象**：尋求健康體態管理、希望擺脫熱量焦慮、需結合生理週期規畫飲食的女性族群。
* **核心理念**：如鳥般指引正確方向，讓健康成為一種美學生活。

## 資訊架構 (Information Architecture)

本站採取「品牌傳遞」與「數位工具」並行的架構，旨在將新用戶轉化為訂閱會員。

| 模組類別 | 包含頁面 / 功能 |
| :--- | :--- |
| **品牌入口** | 品牌 Landing Page、關於朝朝 (創辦人理念)、服務條款、隱私權政策。 |
| **數位工具 (會員專屬)** | 飲食記錄系統、份量查詢資料庫、生理期追蹤月曆、體重體脂圖表分析。 |
| **社群互動** | 會員專屬聊天室、正能量分享頻道。 |
| **個人帳務** | 會員中心、訂閱狀態管理、個人資料設定。 |

## 核心功能設計 (Core Features & Logic)


### **A. 數據驅動的飲食監測系統**
* **智慧份量轉換**：打破傳統熱量計算的焦慮，系統將食物輸入轉化為「六大類份量」管理，讓使用者更直觀地掌握營養均衡。
* **份量查詢資料庫**：內建專業營養團隊開發的資料庫，提供即時、科學化的攝取參考。

### **B. 生理週期體態整合 (Women-Centric Logic)**
* **生理/體態重疊圖表**：將生理期與體重/體脂波動圖表結合，幫助使用者識別「生理性水腫期」與「黃金瘦身期」。
* **智慧預警機制**：在生理期前夕提供飲食與情緒指引，降低因荷爾蒙導致的挫折感。

### **C. 個性化減重計畫 (Smart AI Analysis)**
* **動態計畫生成**：根據會員輸入的初始數據 (BMR/TDEE) 與每日記錄，動態調整推薦的攝取配比，實現精準減重。

## 使用者流程設計 (Key User Flows)

> **流程一：智慧飲食記錄**
> 1. **入口**：會員儀表板 → 點選「紀錄今日餐點」。
> 2. **輸入**：透過「份量查詢」搜尋食物名稱或自定義攝取量。
> 3. **分析**：系統自動換算份量並同步至「當日營養配比」進度條。
> 4. **回饋**：完成紀錄後，系統提供視覺化的達標圖案，提升成就感。

> **流程二：生理期與體重協同追蹤**
> 1. **註記**：使用者於「生理期小月曆」點擊經期起始日。
> 2. **同步**：系統自動在「體重趨勢圖」中標註生理期區間。
> 3. **衛教**：在圖表下方提供「經期體重波動說明」，緩解使用者對體重短暫回升的焦慮。

## 困難及解決

### 會員儀表板的載入時間過久

發現是將儀表板元素拆成不同元件時，資料獲取的方法是寫在各個元件內部，導致有部分api重複調用
以及月曆非同步邏輯沒寫好，在切換日期或是初始化時是要等飲食紀錄&體重體脂&預約諮詢的資料都取得後才會正常顯示，而且在切換月份也會去打所有api（過去時間不會有預約諮詢提醒、未來時間不會有飲食記錄&體重體脂），其實在前端就應該直接擋掉。

後續將變數往上提取到容器元件，其他子元件統一從這個變數提取各自要的資訊，並且完善非同步操作，可以先顯示的資料就先呈現給使用者，月曆部分則是先整理情境，分成過去、當月、未來三個維度，以及飲食記錄、體重體脂、預約諮詢三個面向的資料，把不合理的邏輯直接從前端擋掉，避免多餘的api操作。

### 飲食紀錄的巢狀頁面結構複雜，參數容易丟失或難以管理

一開始分成三個獨立元件發現有這個問題，後來將單日頁面與餐期頁面塞進列表頁面，讓 provide inject 相關的方法可以使用，統一在列表頁定義所有可能的變數與方法再整包傳下去，子頁面有需要就提取需要的變數及方法，並以日期、餐期id、名稱作為path與query string，輔助子頁面與麵包屑也能參考這些參數來定位。

### 飲食紀錄的餐期頁面儲存會產生重複紀錄

這個問題困擾很久，因為以正常流程run起來很難發現，後來解決方法是在每個步驟的所有方法都設定console.log的節點去一一排查，才找到問題點是另一個版本（有的手機能用，有的不能用，因此設定兩個上傳套件，但本地端都是跑其中一個套件測試而已）的上傳圖片套件，上傳圖片時沒有把相關判斷變數改成提交中，後續也針對這一點去優化調整提交中則禁用其他按鈕，避免預期外的操作。

### 關鍵字搜尋時，使用組合式的輸入法會直接觸發搜尋

搜尋這邊是以判斷停止輸入後直接觸發搜尋邏輯，但組合式的輸入法，比如說注音，會在還沒按下組合好的字時直接觸發搜尋（會直接以注音符號搜尋），後來針對Input元件增加 composition相關的事件監聽，與判斷enter的使用時機來解決。

### 聊天室功能元件共用與狀態管理

一對一聊天和社群聊天室兩個功能有多個元件可以共用，但抽出共用元件時難以管理兩個功能的變數傳遞。後來用物件把多個變數包裝在一起傳遞給共用元件，並透過監聽保持props和內部狀態同步。
